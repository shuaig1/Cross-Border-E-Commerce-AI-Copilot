# RAG 检索增强生成优化方案

## 当前架构

```
JSON 商品数据 → text2vec-base-chinese 嵌入 → ChromaDB → 向量语义搜索 → LLM 回复
```

检索链路：`collection.query(query_texts=[query], n_results=3)` → 返回 Top-3 → Agent 组织回复

---

## 一、检索质量

### 1. 混合检索（Hybrid Search）

**现状**：纯语义向量搜索，对精确关键词（SKU 编号、型号、规格参数）匹配能力弱。

**问题**：用户搜 "20000mAh 充电宝"，语义搜索可能返回不相关的电子产品，而精确匹配 "20000mAh" 才是用户意图。

**方案**：
- 引入 BM25 关键词检索（可用 `rank_bm25` 库），对商品标题和描述建立倒排索引
- 向量检索 + BM25 检索并行，结果通过 RRF（Reciprocal Rank Fusion）融合排序
- 也可使用 ChromaDB 自带的 `where` 条件 + 向量搜索的混合模式

**收益**：补齐关键词精确匹配短板，SKU / 品牌名 / 规格参数搜索准确率大幅提升。

---

### 2. 相似度阈值过滤

**现状**：`n_results=3` 固定返回 3 条，无论相关性多低。搜 "宠物智能喂食器" 时即使最相关商品相似度只有 0.3，也照返不误。

**问题**：用户看到毫不相关的商品推荐，体验极差。

**方案**：
```python
results = collection.query(query_texts=[query], n_results=3, include=["documents", "metadatas", "distances"])
# 过滤掉 distance > 阈值的低相关结果
threshold = 1.5  # cosine distance，越小越相关
filtered = [
    (doc, meta) for doc, meta, dist in zip(docs, metas, distances)
    if dist < threshold
]
```

**收益**：成本极低（一行判断），直接解决"搜到无关商品"的核心痛点。建议 P0 先做。

---

### 3. 重排序（Reranking）

**现状**：向量检索返回的 Top-3 排序不够精准，语义相近但非用户意图的商品可能排在第一位。

**方案**：
- 引入 Cross-Encoder 重排序模型（如 `BAAI/bge-reranker-large` 或 `BAAI/bge-reranker-v2-m3`）
- 检索阶段召回 Top-10 候选，Reranker 精排后取 Top-3 返回
- 也可用 LLM 做 Listwise Rerank（让 DeepSeek 自己排），但延迟更高

**收益**：检索准确率显著提升，用户意图匹配更精准。

---

### 4. 查询改写（Query Rewriting）

**现状**：用户原始输入直接用于向量搜索。"有没有便宜点的运动裤" 整句话的嵌入向量可能偏离核心语义 "运动裤"。

**方案**：
- 在 Agent 侧增加查询改写步骤，提取核心类目 + 偏好
- 示例：输入 "有没有便宜点的运动裤" → 改写为 "运动裤 低价"
- 可通过 LLM 做 one-shot query rewrite，或使用 Query2Doc 等轻量方案

**收益**：处理口语化、长尾查询，减少噪声词对嵌入的干扰。

---

### 5. 多轮对话查询补全

**现状**：用户先问 "有跑鞋吗"，再问 "便宜一点的"——第二句缺少主语，向量搜索直接搜 "便宜一点的" 大概率跑偏。

**方案**：
- 利用现有 `chat_history`，在做工具调用前先让 LLM 结合历史补全查询
- 或直接用历史上下文拼接完整查询后再做向量搜索

**收益**：多轮对话场景下的检索准确率提升。

---

## 二、数据层

### 6. 元数据过滤

**现状**：`metadatas` 中存储了 `title`、`price`、`url`，但检索时完全没有利用 price 做过滤。

**问题**：用户问 "50美元以下的跑鞋" 时，向量搜索可能返回 $89.99 的商品。

**方案**：
```python
# 先用 LLM 提取价格/类目等结构化约束
# 再用 ChromaDB 的 where 条件过滤
results = collection.query(
    query_texts=[query],
    n_results=3,
    where={"price": {"$lt": 50}}
)
```
> 注意：ChromaDB 的 `where` 过滤要求字段在 metadata 中，且是等值/范围查询，需要提前规划 metadata schema。

**收益**：价格筛选、品类筛选等结构化查询落地。

---

### 7. 多数据源支持

**现状**：`init_vector_store()` 只读取 `product_vector_rag_store.json` 一个文件。

**方案**：
- 支持从多个 JSON 文件导入
- 支持从 SQLite 数据库（已有 `DataManager`）导入
- 未来可扩展支持 Shopify API 在线商品入库

**收益**：数据来源灵活扩展，不依赖单一文件。

---

### 8. 增量更新 & 脏数据清理

**现状**：重启时 `upsert` 全量数据。JSON 中删除的商品在 ChromaDB 中仍存在。

**方案**：
- 每次 upsert 后，对比 JSON 中的 ID 和 ChromaDB 中的 ID
- 删除 ChromaDB 中存在但 JSON 中已不存在的商品
- 也可提供 API 端点手动触发 re-index

**收益**：删除的商品不会再出现在搜索结果中。

---

## 三、嵌入模型

### 9. 嵌入模型升级

**现状**：`shibing624/text2vec-base-chinese`，768 维，中文短文本表现尚可。

**不足**：
- 中英混合商品标题效果下降
- MTEB 中文榜单排名靠后
- 不支持长文本（max 512 tokens）

**备选方案**：

| 模型 | 维度 | 特点 |
|---|---|---|
| `BAAI/bge-large-zh-v1.5` | 1024 | MTEB 中文榜单第一梯队，中英混合好 |
| `BAAI/bge-small-zh-v1.5` | 512 | 轻量版，速度快 3-5x，效果接近 large |
| `moka-ai/m3e-base` | 768 | 中文社区热门，中英混合优于 text2vec |
| `intfloat/multilingual-e5-base` | 768 | 多语言场景，纯英文商品也可覆盖 |

切换只需改 `model_name`：
```python
embedding_fn = embedding_functions.SentenceTransformerEmbeddingFunction(
    model_name="BAAI/bge-large-zh-v1.5"
)
```
> 切换后需删除 `chroma_db/` 目录重建向量库。

---

## 四、Agent 层

### 10. Prompt 优化

**现状**：System prompt 只说"用自然语言回复"，没有利用检索返回的商品结构。

**方案**：
- 在 prompt 中明确要求 LLM 列出候选商品、对比关键差异、给出推荐理由
- 当检索结果为空时，引导用户提供更多信息而非直接说"未找到"
- 让 LLM 在回复中包含商品链接和价格

**收益**：回复更结构化、更有销售引导力。

---

### 11. 检索失败降级策略

**现状**：向量搜索返回空或低相关结果时，LLM 只能说"未找到"。

**方案**：
- 降级到 SQLite 全量商品列表（`search_products_local_database`），展示品类目录
- 引导用户提供更多关键词
- 记录未命中 query，定期分析后补充商品数据

**收益**：即使搜不到目标商品，用户也不会空手而归。

---

## 五、可观测性

### 12. 检索质量监控

**现状**：无任何日志，检索效果全靠人工感觉。

**方案**：
- 记录每次查询的 query、返回商品、相似度分数、最终回复
- 统计未命中率、平均相似度
- 定期人工抽检或建立评测集（query + 期望商品），量化检索准确率

**收益**：数据驱动优化，知道改了什么、效果变好还是变差。

---

## 优先级排序

| 优先级 | 优化项 | 改动量 | 收益 |
|---|---|---|---|
| **P0** | 相似度阈值过滤 | 极小 | 直接解决无关商品问题 |
| **P0** | 混合检索 | 中 | 补齐精确匹配短板 |
| **P1** | 元数据过滤（价格/品类） | 中 | 结构化查询落地 |
| **P1** | 重排序 | 中 | 检索准确率提升 |
| **P1** | 检索失败降级策略 | 小 | 用户体验兜底 |
| **P2** | 查询改写 | 中 | 处理口语化输入 |
| **P2** | 嵌入模型升级 | 中 | 长期收益，需重建向量库 |
| **P2** | 多轮对话补全 | 中 | 多轮场景可用性 |
| **P3** | 增量更新 & 脏数据清理 | 小 | 数据一致性 |
| **P3** | 多数据源支持 | 大 | 架构扩展性 |
| **P3** | 检索质量监控 | 大 | 长期迭代基础 |

---

## 实施建议

1. **先做 P0**：相似度阈值（10 分钟）和混合检索（半天），这两个改动直接解决当前最痛的问题
2. **再做 P1**：重排序、元数据过滤、降级策略，把检索链路从 60 分提到 80 分
3. **P2/P3 按需推进**：取决于商品库规模和用户量增长
